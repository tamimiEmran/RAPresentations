Prompt Summary

Role Definition

The model is positioned as a smart-grid security and non-technical-loss expert.

  • Domain: smart meters and energy theft.
  • Focus: consumption-pattern manipulation.
  • Assumption: normal reference week is clean.

Input Contract

Each sample contains two images in a fixed order.

  • Image 1: one normal reference week.
  • Image 2: one query day from the same meter.
  • Metadata adds season, weekday, date, and IDs.

Attack Definitions

The prompt defines the retained attack scenarios with behavior and likely causes.

  • Magnitude-reduction attacks: 1, 3, 4, 6, 7.
  • Pattern-altering attacks: 5, 10, 12.
  • Zero/flat attacks: 0, 2, 9, IEEE.

Task Sequence

The model must decide in order, not jump straight to an attack label.

  • Classify original versus manipulated.
  • If manipulated, identify one most likely attack.
  • Explain why the top alternatives were ruled out.

Output Contract

The response format is sectioned text rather than strict JSON.

  • Classification: Original or Manipulated.
  • Identified Attack: most likely, confidence, reasoning.
  • Attack behavior explanation: manipulation, impact, causes.

Evaluation Hooks

The output is parsed into fields for later metric computation.

  • Binary label drives balanced accuracy.
  • Most Likely drives multiclass metrics.
  • Explanation text drives BERT-F1 per class.

Partial Results Snapshot

Metric Order

  • Balanced accuracy first: binary original/manipulated score computed as the average of original recall and manipulated recall.
  • Multiclass second: per-class precision, recall, and F1 over original plus retained attack classes, using the parsed most-likely label.
  • BERT-F1 third: per-class semantic similarity between the generated explanation and that class definition from the prompt.
  • Current BERT-F1 method: extracted Attack_Behavior_Explanation only, bert-base-uncased token matching, no IDF weighting, 512-token truncation per field.
BERT-F1 is computed only on rows with a successful API response, no parse error, and a non-empty parsed Attack_Behavior_Explanation field. Original rows remain N/A unless a separate original-behavior reference text is defined.

Thinking excerpt

So, let's tackle this problem step by step. First, I need to compare the normal week plot with the query day plot to see if there's manipulation.

First, look at the normal week plot. It's a week of data, with x-axis as 15-min intervals up to 700. The consumption shows typical daily patterns: peaks during the day, lower at night. There are recurring patterns every day, with high consumption in the morning, then a drop, then another peak.

[...]

So the manipulation is shifting the peak time. Attack 10 is designed for shifting consumption to off-peak periods, which fits. The query day's total might be similar, which fits Attack 10. Attack 12 would have a different total if it is a lower-consuming user, but the query day's total seems similar.

So the most likely is Attack 10.

Final output

Classification:
- Manipulated

Identified Attack:
- Most Likely: Attack 10
- Confidence: High
- Reasoning: The query day exhibits a reversed temporal pattern compared to the normal week, shifting the typical afternoon/evening consumption peaks to an early-morning peak. This matches Attack 10's definition of reversing the consumption pattern to shift high consumption to off-peak periods without reducing total energy use. Attack 12 was ruled out because it involves replacing with a lower-consuming user's pattern, but the query day's magnitude and total consumption area appear similar to normal, not reduced. Attack 4 was ruled out because it reduces consumption over a period, but the query day's total consumption is consistent with normal behavior.

Attack_Behavior_Explanation:
- The temporal sequence of consumption values is reversed while preserving the total daily consumption amount.
- The magnitude remains similar to normal, but the trend is altered because the timing of peaks shifts from typical daytime/evening hours to early morning.
- Possible cause: false data injection that reorders or inverts the time-series data before transmission.

Byzantine-robust aggregation · early understanding

When does a probabilistic overlay help a robust aggregator?

An overlay scores each worker "honest vs. malicious" and re-weights. The question is simply: when is that worth adding?
From my current understanding: it cannot beat the worst case — but there is a measurable separation power, with a threshold above which an overlay helps any robust aggregator, and below which it does not pay.
Progress notes · 2 June 2026
The setup

A robust aggregator, with a probabilistic overlay

  • In federated learning the server aggregates updates from many workers; a malicious Byzantine minority can poison the result.
  • Robust aggregators (geometric median, coordinate-wise median, trimmed mean, …) discount a minority by design — they down-weight outliers instead of averaging them in.
  • A probabilistic overlay: a layer that scores each worker "honest vs. malicious" and re-weights — abstractly, any such layer.
  • The question: does the overlay help — and exactly when?
The shape of the answer

Two questions, stated up front

Worst case — can an overlay lower the floor a robust aggregator already has?
No
Typical case — when does it help, and which aggregators?
Above a threshold

From my current understanding — the rest is just these two, made precise.

Claim 1 · the worst case

A probabilistic overlay does not improve the floor

In the worst case, no overlay beats the robust aggregator's floor.
  • The robust aggregator already discounts the minority. An overlay's only extra job is discrimination — telling honest from malicious.
  • But the worst case is exactly when that is impossible: a perfect mimic — or honest heterogeneity large enough to swallow the attack — makes honest and Byzantine statistically identical. No score can separate them.
  • So the overlay's only lever is gone, and the worst-case floor Ω(δζ²) (in heterogeneity ζ) stands.
  • This floor is a known result (Karimireddy–He–Jaggi 2022; Allouah et al. 2023). We build on it — we do not beat it, and we are careful not to claim we do.
Claim 2 · the lens

Separation power: how distinguishable the overlay can make them

What it is an overlay reduces each worker to a score. Its separation power = how little the honest and Byzantine score distributions overlap (separation = 1 − overlap A). Method-agnostic.
The two rows top = the honest (blue) and Byzantine (red) gradient clouds; bottom = their scores, with the shaded overlap = A.
Left → right as the clouds pull apart, the scores separate and the overlap shrinks — more separation power. The worst case (Claim 1) is the far-left edge: zero separation.
Two ways left you drift toward "mimic" when the attack is subtle or when honest heterogeneity is large enough to swallow it.

Separation here is the overlay's view (how tell-apart-able the scores are), drawn small to stay on screen — a separate axis from the attack's raw magnitude (how far a bad gradient pulls).

Claim 2 · the threshold

There is a separation threshold S*

Above a threshold separation S*, an overlay helps the robust aggregator. Below it, it does not pay — and an unguarded overlay can actively hurt, by amplifying a Byzantine that fooled it.

A framework claim: that such a threshold exists and governs whether an overlay pays is my working hypothesis. Its exact value — and that it is the same across aggregators — are what I list as open.

Why any robust aggregator

The weights set the effective Byzantine mass — and every robust aggregator degrades with it

Above S*

Informative weights cut the effective mass — δ_eff < δ. Every weighted robust aggregator (each with a breakdown point in δ_eff) improves toward its floor.

Below S*

Uninformative (mimic) weights → δ_eff ≈ δ → no gain, graceful to the floor: it doesn't pay — but a robust combiner does not break.

Genuine harm needs more: a non-robust combiner (a mean), or an adversary that fools the overlay into up-weighting Byzantine (δ_eff ↑). The wrapper guards both.

The overlay changes only the weights a robust aggregator uses, not which aggregator it is — and every robust aggregator degrades with the effective Byzantine mass δ_eff = W_B/(W_G+W_B). That the same threshold works for all of them is the conjecture; the breakdown-point mechanism motivates it.

Evidence · the two mechanisms & a safety guarantee

Below S* it can hurt; above S* it helps — and the downside is bounded

One robust base, sweeping separation power (left = mimic, right = distinguishable). An unguarded overlay (red) blows up at low separation — the hurts; a robust / wrapped overlay (blue) gains at high separation — the helps. A trust-region wrapper caps the downside at a tunable τσ̂ — a deterministic, proved guarantee: ‖x̂ − x̄‖ ≤ ‖base − x̄‖ + τσ̂. (This shows the mechanisms behind the threshold, not the threshold itself.)
Where this points · the research questions

Two questions make "will an overlay help here?" answerable

An overlay helps a robust aggregator only when it can tell honest workers from malicious ones well enough — its separation power. So the work reduces to two clean questions:

Q1 · how much do you get?

For a given robust aggregator + a given probabilistic method, parametrize the separation power they produce — as a function of the attack, the data heterogeneity, and the dimension.

→ predict the separation a concrete pairing supplies.

Q2 · how much is enough?

Parametrize the threshold S* — the separation power above which an overlay helps and below which it doesn't — as a function of the aggregator (its breakdown point) and the Byzantine fraction.

→ predict how much separation a setting needs.

An overlay helps exactly when what you get > what you need (separation power > S*).

Settled so far: the worst-case floor (known) and the bounded-harm guarantee (proved). These two questions are the open part.

From my current understanding

Where this leaves us

  • A probabilistic overlay does not lower the worst-case floor.
  • But there is a measurable separation power with a threshold S*: above it, an overlay helps any robust aggregator; below it, it does not pay — and unguarded, it can hurt.
  • A trust-region wrapper makes the downside bounded either way.
The shift: this turns "should I add a probabilistic layer?" into a measurable question — estimate the separation power and compare it to S*.

How it's made

60 workers, 21 Byzantine (35%) that push the same large shove. An overlay scores each worker; we sweep one knob — the score separation — and average 120 runs per point. x = separation power (left = mimic, right = distinguishable); y = distance from the true honest average (lower is better). The dotted vertical marks where the robust overlay first strictly beats the base.

Black · Base — the robust aggregator alone: robust z-score the scores (median/MAD), reject z>2, geometric median of survivors.
Red · Unguarded overlay — soft weight πᵢ = e−½z₊² on everyone, then a belief-weighted mean (draggable → blows up).
Blue · Robust overlay — same weights π, but a belief-weighted geometric median, argmin_y Σ πᵢ‖y−xᵢ‖ (not draggable).
Green/Purple · Wrapped (τ) — take the unguarded overlay, pull it back into a ball of radius τσ̂ around Base (σ̂ = ‖GM − coord-median‖, computed from data — no ground truth).

What is proved, what is a framework, what is open

Proved

  • The worst-case floor (known result)
  • The safe-wrapper bounded-harm inequality (deterministic)
  • At the mimic limit, no overlay can help

Framework / motivated

  • Separation power as the governing lens
  • The threshold S* itself
  • "Helps any robust aggregator" (mechanism argument)

Open

  • Pinning S* (the Q2 threshold)
  • The exact help / hurt rate
  • Validation across many aggregators

Firm on the floor and the bounded-harm guarantee; humble on one thing — the universality of the threshold across aggregators.
